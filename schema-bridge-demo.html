Section 1

The Gap — Understanding the Problem

This is the setup: Fabric Git integration is strong, but Lakehouse schema serialization is the blind spot. The goal in this section is to make the customer say, “Yes, that is exactly what we’re seeing.”

1
1a

What Fabric Tracks in Git — the Good News

Lead with the positive: Fabric already tracks most solution artifacts extremely well. This helps the audience understand that the challenge is specific, not a blanket critique of Fabric.

✅

Notebooks

Full definition and code are serialized to Git, so notebook logic can be reviewed, versioned, and promoted cleanly.

✅

Data Pipelines

Pipeline definitions are stored as JSON, which means changes are visible in diffs and ride through CI/CD.

✅

Semantic Models

TMDL gives a text-based representation that works naturally with Git reviews and release pipelines.

✅

Reports

Report artifacts are serialized, enabling source control, branching, and controlled promotion.

✅

Warehouse Objects

Warehouse tables, views, procedures, and functions are exported as a SQL project, so the DDL is natively trackable.

✅

Spark Environments

Environment definition is serialized, which supports repeatable package and configuration promotion.

✅

Dataflows Gen2

Dataflow definitions are also tracked, keeping transformation logic under source control.

✅

Overall Story

For most Fabric artifacts, Git really can be the source of truth. That is why the Lakehouse gap stands out so sharply.

1b

What Fabric Does Not Track — the Lakehouse Gap

This is the alarming part. When schema is authored directly inside the Lakehouse experience, the structure is not serialized to Git. The customer must visually feel the difference between “artifact tracked” and “schema missing.”

❌

Lakehouse Tables

Table definitions are not serialized. Git does not receive the CREATE TABLE statement or any equivalent schema artifact.

❌

Lakehouse Views

Views created in the Lakehouse SQL experience are not materialized into text files that can be promoted through Git.

❌

Lakehouse Schemas

Schema objects themselves are not serialized, so environment parity cannot be inferred from Git alone.

❌

Column Definitions

Column names, data types, defaults, nullability, and constraints are not tracked as code when created in the UI.

❌

Stored Procedures

Stored procedure definitions are not natively serialized from the Lakehouse experience for Git-based promotion.

❌

Table Data

Data itself is never tracked in Git—which is expected—but it makes the missing structural DDL even more important.

❌

Files / Folders in OneLake

Files and folder structures in OneLake are not serialized as Git artifacts through the Fabric Lakehouse Git integration layer.

⚠️

Net Result

The workspace can look “Git connected” while the most critical data-structure changes are invisible to source control.

1c

What Is Actually Tracked for a Lakehouse?

Only the Lakehouse container metadata is serialized. This is the distinction that matters: Fabric can recreate the object shell, but not the SQL DDL living inside it.

Display Name

The Lakehouse item name is tracked, which is enough to recreate the artifact identity in the workspace.

GUID / Item Metadata

The container identity and item-level metadata are stored, which helps deployment pipelines understand the artifact shell.

OneLake Shortcuts

Shortcut definitions can be serialized, so references to external data locations may appear in Git.

Data Access Roles (Preview)

Role metadata may be tracked, but this still does not include actual table DDL or relational structure.

Default dbo Schema

The default schema context is represented, yet the objects inside that schema are not exported.

What Never Shows Up

The CREATE TABLE, ALTER TABLE, CREATE VIEW, column names, data types, and constraints never become Git files on their own.

Critical distinction: when a developer creates or alters a table in the Dev Lakehouse UI, the Git commit contains only the Lakehouse container metadata. The actual DDL is not serialized, so UAT and Prod receive the shell, not the structure.
1d

Why This Is a Critical Problem

The failure mode is deceptively simple: deployment succeeds from an artifact perspective, but the data structures the downstream solution depends on are missing.

Developer creates table in Dev Lakehouse UI
↓
Commits to Git → only container metadata committed, NO table DDL
↓
CI/CD deploys to UAT → Lakehouse container exists but TABLES ARE MISSING
↓
UAT is broken — no tables, no views, no schemas for downstream artifacts
↓
Production is also broken if the same promotion path is used
This is exactly the problem Digital Realty is experiencing. Their pipelines deploy almost everything correctly, but the Lakehouse table structures do not come along for the ride.
1e

Impact Analysis

Once the schema is absent in UAT or Prod, every downstream artifact built on top of the Lakehouse becomes fragile.

🧩

Data Pipelines Break

Pipelines reference tables that do not exist in UAT or Prod, so refreshes fail or land in the wrong structure.

📊

Semantic Models Break

Models built on Lakehouse tables cannot bind to their expected source objects when those tables never deploy.

📉

Reports Break

Reports connected to semantic models cascade into failure because their upstream model or Lakehouse source is missing.

📓

Notebooks Break

Notebook SQL queries and Spark reads hit “Table not found” errors even though the workspace and Lakehouse artifact appear to be present.

🕰️

No Schema Versioning

The team cannot answer who changed which columns, when the change happened, or what version to roll back to.

🌡️

Environment Drift

Dev, UAT, and Prod drift silently because UI-authored changes never move through the same promotion mechanism as the rest of the solution.

🛡️

Compliance Risk

There is no durable audit trail of structural changes, which weakens governance, reviewability, and operational accountability.

💸

Release Confidence Drops

Teams stop trusting deployment pipelines and reintroduce manual steps, which slows delivery and increases incident probability.

🔁

Rollback Becomes Guesswork

Without versioned schema files and migration history, restoring a prior state becomes tribal knowledge instead of an engineered process.

Sound bite to use: “Digital Realty’s deployment pipeline is not failing because Fabric cannot promote artifacts. It is failing because the most important Lakehouse structural changes never become artifacts in the first place.”
Section 2

The Bridge — Our Solution

Schema Bridge adds the missing schema-as-code layer around the Lakehouse. It captures live structure into Git, detects drift, promotes versioned migrations, and keeps an auditable history table inside the Lakehouse itself.

2
2a

Solution Overview — “Schema Bridge”

Think of Schema Bridge as a four-pillar operating model around the Lakehouse SQL endpoint. It does not replace Fabric; it fills the serialization and promotion gap until Lakehouse Git support becomes complete.

Schema Bridge Architecture
📸 CAPTURE

Schema Extractor

  • Reads Lakehouse SQL endpoint metadata
  • Reconstructs DDL for tables, views, procedures
  • Writes Git-friendly SQL files and manifest hash
Runs on: GitHub Actions or ad hoc terminal execution
🔍 DETECT

Drift Detector

  • Compares Git export vs live Lakehouse
  • Highlights added, removed, and changed objects
  • Flags manual UI changes before they reach production
Runs on: Daily CI job or manual validation
🚀 PROMOTE

Migration Runner

  • Executes versioned SQL scripts in order
  • Applies schema changes consistently across environments
  • Supports Fabric notebook execution through REST API
Runs on: Fabric notebook triggered by CI/CD
📋 AUDIT

History Table

  • Stores what ran, when, by whom, and with what outcome
  • Provides operational proof and rollback clues
  • Supports time-travel-oriented governance patterns
Lives in: Lakehouse Delta table _migration_history
Git becomes the schema source of truth Dev/UAT/Prod stay structurally aligned Manual UI changes become detectable exceptions Warehouse remains the preferred Gold-layer destination
2b. Pillar 1 — Schema Extractor (📸 CAPTURE) Reads SQL endpoint → writes DDL to Git

What it does

  • Authenticates to the Lakehouse SQL Analytics Endpoint using Azure AD / Entra ID.
  • Queries catalog views such as INFORMATION_SCHEMA.TABLES, INFORMATION_SCHEMA.COLUMNS, and sys.sql_modules.
  • Reconstructs executable DDL because Fabric does not emit Lakehouse SQL as Git-native files.
  • Writes clean output folders that Git can diff, review, and promote.
  • Produces a manifest.json with counts, object names, and a hash for drift detection.
Message to customer: “We are not screen-scraping the UI. We are using the same SQL endpoint metadata your team already trusts.”

2c. Pillar 2 — Drift Detector (🔍 DETECT) Finds manual changes before they become incidents

What drift detection looks for

  • New tables created manually in the Lakehouse UI
  • Columns added or removed outside the Git workflow
  • Type changes, nullability differences, and altered definitions
  • Views or procedures that exist live but are missing from Git
  • Objects in Git that vanished from the target environment

In the repo, scripts\drift_detector.py renders text, JSON, or Markdown and is ideal for scheduled verification or CI comments.

Operational pattern

  • Run daily against the live Dev Lakehouse and the Git export.
  • If drift is found, fail the job and create a GitHub Issue or Azure DevOps work item.
  • Team decides: capture the change intentionally, convert it into a migration, or revert it.
  • Promotions are blocked until Git and live state are reconciled.
Safety net framing: drift detection does not prevent bad changes from happening; it prevents them from remaining invisible.

        run: echo "Create GitHub Issue / Azure DevOps work item here"
2d. Pillar 3 — Migration Runner (🚀 PROMOTE) Schema-as-code promotion across environments

How promotion works

  • Developers write versioned SQL scripts such as v001_baseline_tables.sql and v002_capacity_and_sustainability.sql.
  • Scripts are reviewed in pull requests exactly like application code.
  • A Fabric notebook discovers pending migrations, checks history, and runs them in order.
  • Each environment receives the same ordered set of changes, eliminating “it worked in Dev only” schema drift.
  • The notebook records outcomes in _migration_history.

Repository-backed proof points

  • lakehouse-migrations\migrations\v001_baseline_tables.sql creates data_centers, power_consumption, and cooling_metrics.
  • lakehouse-migrations\migrations\v002_capacity_and_sustainability.sql adds capacity_utilization, sla_incidents, and new sustainability columns.
  • lakehouse-migrations\migration_runner.py already contains discovery, execution, and audit logic.
  • pipelines\.github\workflows\deploy-schema.yml and pipelines\azure-pipelines.yml already show promotion automation.

Actual notebook pattern from migration_runner.py

migrations = discover_migrations(base_path=migrations_path, selected_type=migration_type)
applied = get_applied_migration_ids()
pending = [migration for migration in migrations if migration.migration_id not in applied]

for migration in pending:
    start = time.time()
    try:
        execute_migration(migration)
        record_migration_result(
            migration_id=migration.migration_id,
            status="success",
            duration_seconds=time.time() - start,
        )
    except Exception as exc:
        record_migration_result(
            migration_id=migration.migration_id,
            status="failed",
            duration_seconds=time.time() - start,
            error_message=str(exc),
        )
        raise

2e. Pillar 4 — Audit Trail (📋 AUDIT) History, accountability, rollback clues

The audit mechanism

Every Lakehouse should carry its own _migration_history Delta table. That table is the durable log of structural changes that actually ran in that environment.

  • migration_id — e.g., v002_capacity_and_sustainability
  • applied_at — timestamp of execution
  • applied_by — service principal, user, or pipeline identity
  • status — success / failed
  • duration_seconds — performance and troubleshooting signal
  • error_message — failure reason when applicable

Example query pattern

SELECT
    migration_id,
    applied_at,
    applied_by,
    status,
    duration_seconds
FROM _migration_history
ORDER BY applied_at DESC;

-- Because the table is Delta-backed, the history can participate
-- in broader governance and time-travel-oriented operations.
Why this matters to customers: once migrations and history are in place, schema change becomes auditable, reviewable, and explainable. That is the opposite of silent UI drift.
2f. Alternative Recommendation — Use Fabric Warehouse for the Gold Layer Native Git serialization where it matters most for BI

Best-practice architecture recommendation

Use the Lakehouse where Spark and open-format data engineering shine — typically Bronze and Silver. Use Fabric Warehouse for the curated Gold layer, where T-SQL, BI-ready objects, and native Git serialization are strongest.

  • Lakehouse: raw ingestion, cleansing, enrichment, Spark notebooks, Delta processing
  • Warehouse: curated dimensions, facts, views, procedures, BI-ready semantic serving layer
  • Schema Bridge: closes the Lakehouse gap until native serialization catches up
MyWarehouse.SQLProject/
├── dbo/Tables/
│   ├── dim_data_center.sql
│   └── fact_power_usage.sql
├── dbo/Views/
│   └── vw_daily_pue.sql
└── dbo/StoredProcedures/
    └── sp_refresh_facts.sql

These files are natively tracked in Git for Warehouse — no extra bridge is needed for the Gold layer.

Capability Lakehouse Warehouse Recommendation
Container metadata in Git Partial Yes Both are deployable artifacts, but Warehouse is richer.
Table DDL serialized to Git No Yes Use Schema Bridge for Lakehouse; rely on native support for Warehouse.
Views / procedures / functions in Git No Yes Warehouse is the stronger home for Gold-layer relational objects.
Spark-native engineering Yes No Keep heavy engineering and open-format processing in the Lakehouse.
Best fit Bronze / Silver Gold / BI Use both deliberately instead of forcing one to do all jobs.
Need for Schema Bridge Yes No Bridge only where Git serialization is missing.
Customer recommendation sentence: “For Digital Realty, keep Lakehouse for Bronze/Silver engineering, move Gold relational objects into Warehouse where possible, and use Schema Bridge to make Lakehouse schema promotion safe in the meantime.”

Section 4

FAQ / Tough Questions

These are the questions most likely to come from a thoughtful customer. Keep the answers calm, direct, and anchored in what the solution actually does.

4
1. What if someone makes a change directly in the Lakehouse UI?

The drift detector catches it within the next scheduled run — typically within 24 hours — and can create a GitHub Issue or Azure DevOps work item automatically. The team then decides whether to capture that change intentionally, convert it into a proper migration, or revert it.

  • Best short answer: “UI changes are allowed by Fabric, but they stop being invisible.”
  • Best redirect: “Our preferred operating model is still Git-first migrations.”
2. Can we enforce that all schema changes go through Git?

Not natively in Fabric Lakehouse today — there is no hard “read-only for DDL” mode that forces Git-first behavior. But you can get very close operationally.

  • Use Fabric roles to limit who can edit Lakehouse schema.
  • Run drift detection and treat drift as a release blocker.
  • Place Gold-layer relational objects in Warehouse, where Git serialization is native.
3. Why not just use Fabric Warehouse instead?

Warehouse is an excellent answer for Gold-layer BI objects because it supports SQL project export to Git natively. But Lakehouse remains stronger for Spark-based Bronze/Silver engineering, open data processing, and data science workloads.

  • Best short answer: “Use both — Lakehouse for engineering, Warehouse for curated BI.”
  • Schema Bridge is the interim answer where Lakehouse Git serialization is missing.
4. How do we handle rollbacks?

Delta Lake supports time travel and restore patterns. Migration scripts can also include explicit rollback logic when appropriate. The _migration_history table tells you exactly what was applied in each environment and when.

  • Good phrase: “Rollback gets easier once schema is versioned and execution is logged.”
5. What about the fabric-cicd Python library?

It helps automate deployment of Fabric items such as notebooks, pipelines, reports, and semantic models. It does not solve the Lakehouse schema serialization gap by itself.

  • Best short answer: “Schema Bridge is complementary, not competitive.”
6. How long does a migration take to run?

Individual SQL migrations usually run in seconds. End-to-end pipeline duration is typically 5–10 minutes depending on approval timing, notebook startup latency, and any validation steps around the deployment.

7. What if two developers create migrations with the same version number?

That should be caught in pull request review. You can also add a CI validation step to detect duplicate version numbers. The migration runner relies on ordered versioning, so naming discipline matters.

  • Best short answer: “Treat version collisions the same way you would any conflicting release artifact in Git.”
8. Is Microsoft planning to fix this gap?

The Fabric team is aware of the challenge. Warehouse already has mature Git serialization. Lakehouse may improve over time, but Schema Bridge is the practical answer customers can implement now rather than waiting on roadmap timing.

9. Can we use this with Azure DevOps instead of GitHub?

Yes. The repo already includes pipelines\azure-pipelines.yml and the scripts themselves are CI-platform agnostic. GitHub Actions and Azure DevOps can run the same extractor, drift detector, and migration notebook pattern.

10. What about data migration between environments?

Schema Bridge handles DDL — structure, not data movement. For data, use Fabric Deployment Pipelines where appropriate, or run Spark notebooks / copy patterns to seed reference data and environment-specific datasets.

  • Best short answer: “Schema Bridge is for structure; data movement is a separate concern.”
