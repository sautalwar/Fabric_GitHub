The deployment pipeline succeeds. The Lakehouse artifact exists. But the tables inside it do not. Downstream notebooks, pipelines, and reports break.

This is not a bug — it is a serialization gap. Fabric serializes the Lakehouse as an item container but does not export the internal SQL DDL to Git-trackable files. Warehouse does not have this gap — its SQL Project export includes full DDL natively. The Fabric product team is aware of this limitation. See: Lakehouse Git Integration docs

Pillar 1: Schema Extractor

scripts/schema_extractor.py — Reads your Lakehouse tables and writes SQL files to Git

What does it do?

The Schema Extractor connects to your Lakehouse SQL endpoint (the same endpoint your team already queries), reads the INFORMATION_SCHEMA metadata, and reconstructs CREATE TABLE DDL statements into files that Git can version-control. Think of it as "take a snapshot of my database schema and save it as code."

How to run it:

$ python scripts/schema_extractor.py \
    --endpoint "your-lakehouse.sql.azuresynapse.net" \
    --database "DataCenterLakehouse" \
    --output-dir "schema-export" \
    --auth "azure-cli"

What it creates:

📁 schema-export/
  📁 tables/
    📄 dbo.data_centers.sql
    📄 dbo.power_consumption.sql
    📄 dbo.cooling_metrics.sql
    📄 dbo.capacity_utilization.sql
    📄 dbo.sla_incidents.sql
  📁 views/
  📁 procedures/
  📄 manifest.json ← drift hash
Each .sql file contains a complete CREATE TABLE statement with all columns, types, and COMMENT clauses. The manifest.json stores a hash for drift detection.

Example output — tables/dbo.data_centers.sql:

CREATE TABLE IF NOT EXISTS data_centers ( dc_id STRING COMMENT 'Logical primary key for the data center', dc_name STRING COMMENT 'Human-readable name', region STRING, city STRING, country STRING, total_capacity_kw DOUBLE, tier_level INT, status STRING ) USING DELTA;

🗄️ Pillar 1 — Schema Extractor (scripts/schema_extractor.py)

WHY IS IT NEEDED?

"When you connect a Fabric workspace to GitHub, most artifacts serialize automatically — notebooks become .py files, pipelines become JSON, semantic models become TMDL. But Lakehouse TABLE SCHEMAS do NOT serialize. If you create a table called data_centers with 9 columns in your Dev Lakehouse, Git knows NOTHING about it. The Lakehouse folder in GitHub just contains metadata — not the table definitions."

"So if you want to recreate those same tables in UAT or Prod, you have no source of truth. Someone has to manually recreate them or copy-paste CREATE TABLE statements. That's the gap. The Schema Extractor exists to fill this gap — it reads what's actually in your Lakehouse and writes it as SQL files that Git can version-control."

WHO WRITES IT?

"The Schema Extractor is written ONCE by your DevOps or Platform Engineering team. It's a Python script — about 400 lines — that uses pyodbc to connect to the Lakehouse SQL Analytics Endpoint and azure-identity for authentication. Your data engineers never touch it. They just run it (or it runs automatically in CI/CD). Think of it like a utility — the plumbing team builds it, everyone else benefits."

"In this demo, I wrote it with GitHub Copilot in about 15 minutes. Your team doesn't need to write it from scratch — it's already in the repository at scripts/schema_extractor.py, ready to use."

GITHUB COPILOT PROMPT TO GENERATE THE SCHEMA EXTRACTOR:

"If you want to generate this from scratch using Copilot, here's the exact prompt to give in Copilot Chat (Ctrl+Shift+I in VS Code):"

"Write a Python script called schema_extractor.py that connects to a Microsoft Fabric Lakehouse SQL Analytics Endpoint using pyodbc and azure-identity. It should: (1) Accept command-line arguments for --endpoint, --database, --output-dir, and --auth method (service-principal, azure-cli, managed-identity, interactive). (2) Query INFORMATION_SCHEMA.TABLES and INFORMATION_SCHEMA.COLUMNS to discover all user tables. (3) For each table, reconstruct a complete CREATE TABLE IF NOT EXISTS statement with column names, data types, and COMMENT clauses, using USING DELTA syntax. (4) Write each table's DDL to output-dir/tables/schema.tablename.sql. (5) Also handle views (write to views/) and stored procedures (write to procedures/). (6) Generate a manifest.json with table counts, object names, and a SHA-256 hash of all DDL content for drift detection. (7) Use the SQL_COPT_SS_ACCESS_TOKEN = 1256 connection attribute to pass the Azure AD token to pyodbc. (8) Support retry logic and proper error handling. "Copilot will generate 80-90% of this correctly on the first try. You'll need to tweak the ODBC token struct packing (it uses a specific byte format for SQL Server access tokens) and test against your actual endpoint. But the core logic — INFORMATION_SCHEMA queries, DDL reconstruction, file writing — Copilot nails it."

The problem it solves:

Someone opens the Fabric portal, adds a column to a table directly in the UI, and never tells anyone. That change exists in the live Lakehouse but not in Git. 🔍 Pillar 2 — Drift Detector (scripts/drift_detector.py)

WHY IS IT NEEDED?

"Here's the real-world problem: You've set up Git integration, you've got versioned migrations, everything looks great. Then on a Tuesday afternoon, a data engineer opens app.fabric.microsoft.com, clicks on the Lakehouse, and adds two columns to the power_consumption table directly in the UI. They needed it for a quick report. They never tell anyone. They never create a migration file."

"That change now exists in the LIVE Lakehouse but NOT in Git. Your Git repo thinks power_consumption has 8 columns. The live table has 10. This is called SCHEMA DRIFT — and without a detector, it's completely invisible. The next time someone writes a v003 migration, they won't know about those 2 rogue columns, and the migration could conflict or produce unexpected results."

"The Drift Detector exists to make these invisible changes visible — it compares what Git thinks the schema looks like vs what the Lakehouse actually has, and reports every difference."

WHO WRITES IT?

"Like the Schema Extractor, the Drift Detector is written ONCE by your DevOps or Platform Engineering team. It's about 350 lines of Python. It imports the schema_extractor module to do a fresh export of the live Lakehouse, then compares it against the Git-tracked export. Your data engineers never touch it — it runs automatically via the drift-detection.yml GitHub Actions workflow every weekday morning at 6 AM UTC. If drift is found, it creates a GitHub Issue automatically. The team just sees the Issue notification and decides what to do."

"In this demo, the script is already in the repo at scripts/drift_detector.py — ready to use."

GITHUB COPILOT PROMPT TO GENERATE THE DRIFT DETECTOR:

"Here's the exact prompt to give in Copilot Chat (Ctrl+Shift+I in VS Code):"

"Write a Python script called drift_detector.py that compares a Git-exported Fabric Lakehouse schema against either a live Lakehouse or another export directory. It should: (1) Accept --git-schema-dir (path to the Git-tracked schema export), and either --endpoint + --database (for live comparison) or --live-schema-dir (for offline comparison). (2) Load all .sql files from both sides and parse them into a normalized representation (table name, column name, column type, ordinal position). (3) Compare the two snapshots and categorize each table as: ADDED (exists live but not in Git), REMOVED (exists in Git but not live), MODIFIED (columns differ — show which columns were added, removed, or changed), or UNCHANGED. (4) For MODIFIED tables, show a detailed diff: which specific columns were added, removed, or had their type changed. (5) Support output formats: text (for terminal), markdown (for GitHub Issues), and JSON (for programmatic use). (6) Import the schema_extractor module to do a live extraction if --endpoint is provided. (7) Exit with code 1 if drift is found (for CI/CD gating). Use dataclasses for the comparison objects, proper logging, and argparse for CLI arguments."

"Copilot generates about 85% of this correctly. The main thing you'll adjust is the column-level normalization logic — making sure data type comparisons are case-insensitive and handle aliases (e.g., STRING vs VARCHAR). The overall structure, argparse setup, file loading, and diff reporting all come out clean."

BEST PRACTICES FOR DRIFT DETECTION:

• Run on a schedule, not on every push: Drift detection requires connecting to the live Lakehouse, which costs Fabric capacity. Running it once daily (weekday mornings) is the sweet spot — frequent enough to catch changes quickly, not so frequent that it wastes compute.

• Create GitHub Issues automatically: Don't just log drift to a file — create a GitHub Issue with the 'schema-drift' and 'needs-review' labels. This puts it in front of the team in the tool they're already using.

• Don't auto-fix drift: The detector should REPORT drift, not fix it. The team needs to decide: was this change intentional? If yes, create a migration file to formalize it. If no, revert it by re-running the last known-good migration. Auto-fixing can mask real problems.

• Compare Dev, UAT, AND Prod: Run drift detection against all environments. Dev drift is expected (developers experiment). UAT drift is a yellow flag. Prod drift is a red flag — someone bypassed the pipeline.

• Include the drift report in PRs: Before merging a migration PR, run the drift detector as a PR check. If there's existing drift, the migration author needs to account for it in their script.

• Baseline first: Before enabling drift detection, run the Schema Extractor to create a clean baseline export. Commit it to Git. Then the Drift Detector has something to compare against.

DETAILED WALKTHROUGH:

"The drift detector compares what's in Git (the schema-export directory) with what's live in the Lakehouse. It produces a report showing: MODIFIED (columns added/changed), ADDED (new tables not in Git), REMOVED (tables deleted from live), and UNCHANGED (everything matches).

It can also compare two offline export directories — useful for comparing Dev vs Prod schemas without connecting to both Lakehouses simultaneously.

In CI/CD, the drift-detection.yml workflow runs this every weekday morning at 6 AM UTC. If drift is found, it automatically creates a GitHub Issue with the full report and labels it 'schema-drift' and 'needs-review'. The team gets notified. Manual changes are no longer invisible."

🚀 Pillar 3 — Migration Runner (lakehouse-migrations/migration_runner.py)

WHY IS IT NEEDED?

"Here's the core problem: you have SQL files in Git that define your tables — v001_baseline_tables.sql creates data_centers, power_consumption, cooling_metrics. v002 adds capacity_utilization and sla_incidents. But SQL files sitting in a Git repo don't DO anything. Someone (or something) needs to actually EXECUTE those SQL statements against the Lakehouse to create the tables. That's what the Migration Runner does — it's the bridge between 'schema defined in code' and 'tables actually existing in the Lakehouse.' Without it, your migration files are just documentation. With it, they're executable infrastructure."

"The second problem it solves: CONSISTENCY ACROSS ENVIRONMENTS. Your Dev Lakehouse has 5 tables. You need the exact same 5 tables in UAT and Prod. Without the runner, someone has to manually execute SQL in each environment — which is error-prone and unauditable. The runner automates this: point it at any environment, it figures out what's missing, and applies only what's needed."

WHO WRITES IT?

"The Migration Runner is written ONCE by the DevOps or Platform Engineering team. It's a Fabric notebook — about 320 lines of Python — that runs inside the Fabric Spark environment. It uses spark.sql() for SQL execution and spark.table() for Delta table operations. Your data engineers never modify it. They write migration FILES (the v001, v002 SQL scripts), and the runner takes care of executing them in the right order. Think of it like a database migration tool (Flyway, Alembic, Entity Framework Migrations) — but built specifically for Fabric Lakehouse."

"In this demo, it's already in the repo at lakehouse-migrations/migration_runner.py. You upload it as a notebook to your Fabric workspace, and the CI/CD pipeline triggers it via the REST API."

WHAT VALUE DOES IT BRING?

• Automation: No one manually runs SQL against production. The CI/CD pipeline triggers the runner, and it handles everything.

• Safety: It only runs what's NEW. If v001 and v002 are already applied, it skips them. No risk of re-creating tables or duplicating data.

• Auditability: Every execution is logged — who ran it, when, which version, success or failure. Compliance teams love this.

• Repeatability: Run it once, run it 100 times — same result. If it fails halfway, fix the SQL, re-run it — it picks up where it left off.

• Environment parity: Dev, UAT, and Prod all go through the same runner with the same migration files. No more 'works in Dev, breaks in Prod' because someone forgot a column.

THE THINKING BEHIND EACH STEP:

Step 1 — Discover all migration files: "The runner looks in the lakehouse-migrations/migrations/ directory for files matching the pattern vNNN_description.sql (or .py). It sorts them by version number — v001 before v002 before v003. WHY? Because migrations must run in order. v002 might ALTER a table that v001 created. If you run v002 first, it fails because the table doesn't exist yet. Sorting by version number guarantees the correct execution order."

Step 2 — Query _migration_history: "The _migration_history table is a Delta table that lives inside the Lakehouse itself. It records every migration that has been successfully applied. WHY do we query it? Because we need to know WHAT HAS ALREADY RUN. If this is a fresh environment (e.g., new UAT Lakehouse), _migration_history doesn't exist yet — the runner creates it automatically. If this is an existing environment, it already has rows showing v001 was applied on Nov 15 and v002 on Dec 3. We read this table to build a set of 'already applied' migration IDs."

Step 3 — Filter to pending only: "Now we compare the two lists: (A) all discovered migration files and (B) all already-applied migration IDs. The difference = PENDING migrations. WHY? Because re-running an already-applied migration could cause errors (e.g., creating a table that already exists) or waste time. By filtering, the runner is SMART — it only does work that hasn't been done yet. Example: discovered = [‍v001, v002, v003], already applied = [‍v001, v002], pending = [‍v003]. Only v003 runs."

Step 4 — Execute each SQL via spark.sql(): "For each pending migration, the runner reads the .sql file content, splits it into individual SQL statements (separated by semicolons — handling edge cases like semicolons inside strings or comments), and executes each statement one by one via spark.sql(). WHY one at a time? Because spark.sql() executes a single statement. A migration file like v002 contains 3 statements (CREATE TABLE capacity_utilization, CREATE TABLE sla_incidents, ALTER TABLE power_consumption) — they must be split and run individually. For PySpark migrations (.py files), the runner uses exec() to execute the Python code in the current notebook scope, so it has access to the spark session."

Step 5 — Log result to _migration_history: "After each migration executes (whether it succeeds or fails), the runner writes a row to _migration_history with: migration_id, migration_name, applied_at (timestamp), applied_by (the user or service principal), migration_type (sql or pyspark), status (success or failed), and duration_seconds. WHY? Two reasons: (1) the NEXT time the runner executes, it reads this table to know what to skip, and (2) COMPLIANCE — you have a permanent, queryable audit trail of every schema change ever made to every environment."

WHAT DOES IDEMPOTENT MEAN AND WHY DO WE NEED IT?

"Idempotent means: you can run the same operation multiple times and get the same result as running it once. In practical terms: if you trigger the Migration Runner 5 times in a row against the same environment, the FIRST run applies any pending migrations. Runs 2 through 5 do NOTHING — they see that everything is already applied and skip everything. No errors. No duplicate tables. No data corruption."

"WHY is this critical for CI/CD? Because automated pipelines RETRY. If a GitHub Actions workflow fails halfway (e.g., network timeout), it retries the job. If the runner already applied v001 and v002 before the failure, the retry needs to be safe — it must skip v001 and v002 and only retry whatever failed. Without idempotency, the retry would try to CREATE TABLE data_centers again, get an 'already exists' error, and fail."

"We achieve idempotency through TWO mechanisms working together:"

1. _migration_history tracking: The runner checks which versions are already applied BEFORE executing anything. Applied = skip. Pending = run.

2. CREATE TABLE IF NOT EXISTS: Even if the history check somehow fails (e.g., history table was corrupted), the SQL itself won't error — IF NOT EXISTS means 'create this table only if it doesn't already exist.' Double safety net.

"This is the same pattern used by Flyway (Java), Alembic (Python), Entity Framework Migrations (.NET), Django Migrations (Python), and Liquibase (Java). We're applying an industry-proven pattern to Fabric Lakehouse."

📋 Pillar 4 — Audit Trail (_migration_history)

WHAT IS _migration_history?

"The _migration_history table is a Delta table that lives inside the Lakehouse itself. It is NOT a separate database or external system — it's a regular Delta table sitting right next to your data_centers, power_consumption, and cooling_metrics tables. The Migration Runner creates it automatically on its very first run if it doesn't already exist."

"Every row in this table represents one migration execution. It records: the migration_id (e.g., v002_capacity_and_sustainability), the migration_name (human-readable), applied_at (exact UTC timestamp), applied_by (the user or service principal that triggered the run), migration_type (sql or pyspark), status (success or failed), and duration_seconds (how long it took). This gives you a complete, queryable audit trail of every schema change ever applied to this Lakehouse."

WHY IS IT NEEDED?

"Compliance teams always ask the same question: 'When did this column get added to Production, and who approved it?' Without an audit trail, you're digging through Slack messages, emails, and memory. With _migration_history, the answer is a simple SQL query:"

SELECT * FROM _migration_history WHERE migration_id = 'v002_capacity_and_sustainability'

"Answer: Applied on 2024-12-03 at 14:22:07 UTC, by service principal FabricCICD-Demo (client ID ending in ...a3b7), migration type SQL, status success, took 2.3 seconds. The APPROVAL record is in the GitHub Pull Request that merged v002 into the main branch — complete with who reviewed it, who approved it, and the full diff of what changed."

WHO CREATES IT?

"Nobody creates it manually. The Migration Runner's code includes a CREATE TABLE IF NOT EXISTS _migration_history statement that runs at the very beginning of every execution. First time? Table gets created. Second time? IF NOT EXISTS means it's skipped. This is part of the idempotent design — the runner sets up its own infrastructure."

HOW TO USE IT IN PRACTICE:

• Audit query: SELECT migration_id, applied_at, applied_by, status FROM _migration_history ORDER BY applied_at DESC — shows the complete history, newest first.

• Find failures: SELECT * FROM _migration_history WHERE status = 'failed' — immediately surfaces any migrations that didn't complete successfully.

• Compare environments: Run the same query against Dev, UAT, and Prod _migration_history tables to see if they're in sync. If Prod is missing v003 but Dev and UAT have it, you know the Prod deployment hasn't run yet.

• Compliance reporting: Export the table periodically for SOC 2 or ISO 27001 evidence — it proves that all schema changes went through an automated, auditable pipeline.

WHAT HAPPENS WHEN A MIGRATION FAILS?

"If v003 fails (e.g., a syntax error in the SQL), the runner logs it to _migration_history with status = 'failed'. The CI/CD pipeline sees a non-zero exit code and marks the GitHub Actions run as failed. The team gets a notification. They fix the SQL in v003, push a new commit, and CI/CD re-runs. The runner queries _migration_history, sees v001 and v002 are 'success' (skip them), sees v003 is 'failed' (re-try it), and executes the fixed version. If it succeeds, a new row is added with status = 'success'. Both rows are preserved — the failed attempt AND the successful retry — giving you a complete history of what happened."

KEY TALKING POINT: "This is exactly how enterprise database migration tools work — Flyway, Liquibase, Django, EF Migrations — they all maintain a history table. We're applying the same proven pattern to Fabric Lakehouse, which doesn't have this built in natively."
