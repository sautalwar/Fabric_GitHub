💡 What is a Lakehouse?

Storage: Delta Parquet files in OneLake — open format, no vendor lock-in

Compute: Apache Spark for engineering, SQL endpoint for analytics

Access: Notebooks (PySpark/SQL), SQL endpoint (read-only T-SQL), Power BI (Direct Lake)

Schema: Flexible — schema-on-read with enforcement via Delta Lake

ACID: Full transactional support through Delta Lake transaction log

Best for: Data engineering, ETL/ELT, ML, medallion architecture (bronze/silver/gold)

Git Integration

What Fabric Tracks in Git

Most artifacts are fully version-controlled. But there's one critical gap.

✅ Tracked by Git

NotebooksFull definition + code
Data PipelinesFull definition
Semantic ModelsFull definition
ReportsFull definition
Warehouse objectsTables, Views, Procs, Functions
Spark EnvironmentsFull definition
Dataflows Gen2Full definition

❌ NOT Tracked by Git

Lakehouse TablesNot serialized
Lakehouse ViewsNot serialized
Lakehouse SchemasNot serialized
Column definitionsNot serialized
Stored ProceduresNot serialized
Table dataNever tracked
Files/FoldersNot serialized

⚠ What IS tracked for a Lakehouse?

Only the container metadata: display name, GUID, OneLake shortcut definitions, and Data Access Roles (preview). The actual table DDL (CREATE TABLE, ALTER TABLE, column names, data types) is never serialized to Git.

The Problem

The Lakehouse Schema Gap

Schema changes made in Dev don't propagate to UAT or Prod.

Dev Lakehouse

→
📁
Git Repository
⚠ Only container metadata — no tables

🏗️

❌ Empty — no tables deployed!

🔥 Real-World Impact

Day 1–60

Team builds 30 tables in Dev via SQL endpoint. Commits notebooks to Git.

Release Day

Deploy to UAT via pipeline. All notebooks arrive — but zero tables exist.

Day 61–63

3 days manually recreating tables in UAT. Repeat for Prod.

Month 3

Column added in Dev, missed in Prod. Production breaks at 2 AM.

What Goes Wrong Without a Solution

1

Schema Drift

Dev, UAT, and Prod have different table structures. No source of truth.

2

No Audit Trail

No record of who changed what, when. Compliance risk.

3

No Code Review

Schema changes bypass PR reviews and governance controls.

4

No Rollback

Can't revert a bad schema change through Git — must fix forward manually.

The Solution

Schema-as-Code

Treat your Lakehouse schema like application code. Version it. Review it. Automate it.

🔢 Versioned

Each schema change is a numbered migration file (v001, v002, v003...). Full history in Git with diffs, PRs, and blame.

🔁 Idempotent

Run the migration 10 times — same result. Uses IF NOT EXISTS and a history table to track what's already applied.

🤖 Automated

Push a migration script → CI/CD applies it to UAT → approval gate → applies to Prod. Zero manual clicks.

🎯 This is the industry-standard pattern

The same approach used by Flyway, Liquibase, Django Migrations, Entity Framework Migrations, and Alembic. Proven across millions of production databases worldwide. We're applying it to Fabric Lakehouse.

📓 3. Migration Runner — Deep Dive

Positioning: "This is the execution engine — a parameterized Fabric notebook that knows what has already run and only applies what's new."

1
Open migration_runner.py in Fabric — show the top parameters cell: target_environment = "Dev", migration_type = "sql", dry_run = False.
"The notebook is parameterized. CI/CD passes the target environment — Dev, UAT, or Prod — so the same notebook works everywhere."
2
Walk through the 5-step flow — ① Read parameters → ② Scan migrations/ folder, sort by version number → ③ Query _migration_history Delta table for already-applied migrations → ④ Apply only the pending ones → ⑤ Record each result with timestamp, status, duration.
"It's a 5-step loop: discover, filter, execute, record, report. The history table is the key — it makes this idempotent."
3
Run it live — click Run All in the Fabric notebook. Show the output: banner, discovered migrations, "already applied" vs "applying now", and the final summary with ✅ checkmarks.
"Watch what happens — v001 is skipped because it's already in the history table. Only v002 runs. That's idempotency in action."
4
Query the history table — run SELECT * FROM _migration_history ORDER BY applied_at in the SQL endpoint. Show the audit trail: migration name, environment, status, applied_by, timestamp.
"This is your audit log. Compliance team asks 'when did this column get added to production?' — the answer is right here."
5
Re-run the notebook — run it again to prove idempotency. Output shows "0 migrations pending — environment is up to date."
"Run it again. Nothing happens. Safe to re-run after failures, retries, or just to verify. That's the idempotency guarantee."

📁 File Naming

v001_, v002_, v003_... — version prefix ensures execution order is explicit and deterministic.

🛡️ Idempotency

IF NOT EXISTS on every CREATE. Safe to re-run. No duplicate tables. No errors on retry.

🌐 Dual Format

Every migration available as SQL (simpler) and PySpark (more powerful). Choose based on complexity.

Execution Engine

Migration Runner Notebook

A parameterized Fabric notebook that applies only new migrations. Idempotent. Safe. Auditable.

1

Read Parameters

Target environment (Dev/UAT/Prod) and migration type (SQL/PySpark). Passed by CI/CD or set manually.

2

Scan Migration Files

Reads all migration scripts from the lakehouse-migrations/migrations/ folder. Sorts by version number.

3

Check History

Queries the _migration_history Delta table to find which migrations have already been applied.

4

Apply New Migrations

Runs only the migrations that haven't been applied. Logs each one with timestamp, status, and duration.

5

Record State

Writes each successful migration to _migration_history. Creates a permanent audit trail.

Python — Notebook
# migration_runner.py # Parameterized Fabric notebook # ── Parameters ── target_environment = "Dev" # Dev | UAT | Prod migration_type = "sql" # sql | pyspark # ── Step 1: Get available migrations ── migration_files = get_migration_files( migration_type ) # ── Step 2: Check what's already applied ── applied = spark.sql(""" SELECT migration_name FROM _migration_history WHERE environment = '{}' """.format(target_environment)) # ── Step 3: Find pending migrations ── pending = [m for m in migration_files if m.name not in applied_set] # ── Step 4: Apply each one ── for migration in pending: print(f"Applying: {migration.name}") spark.sql(migration.read_text()) record_migration( migration.name, target_environment, "success" ) print(f"✅ {migration.name} applied") print(f"Done! {len(pending)} applied.")

Architecture

Complete CI/CD Architecture

Two parallel tracks working together for full coverage.

Track 1 Standard Artifacts

Handled by Fabric Git + Deployment Pipelines

  • Notebooks committed to Git from workspace
  • Data Pipelines synced via Git integration
  • Semantic Models and Reports versioned
  • Deployment Pipeline promotes Dev → UAT → Prod
  • Environment-specific configuration rules
  • Built into Fabric — no custom code needed

Track 2 Schema Evolution

Handled by Schema-as-Code + CI/CD Pipeline

  • Migration scripts (SQL/PySpark) in Git
  • PR reviews enforce schema governance
  • Migration runner notebook applies changes
  • CI/CD pipeline triggers notebook via REST API
  • Approval gates for production deployment
  • History table provides audit trail

🏗️ Lakehouse vs Warehouse: When to Use Each

🏗️ Lakehouse

Data engineering, ETL/ELT, ML, semi-structured data, medallion architecture (bronze/silver), Direct Lake for Power BI. Schema tracked via Schema-as-Code.

🏢 Warehouse

BI reporting, curated gold layer, full T-SQL, stored procedures, multi-table transactions. Schema tracked natively by Git integration.

Best practice: Use BOTH together — Lakehouse for engineering, Warehouse for serving. Full schema tracking across the board.
